# v0.2 正式发布前全面治理计划

## Problem Frame

51guapi 功能主骨架已完整（爬取 → 提炼 → 待审 → 导出），测试覆盖扎实。但存在两个 P0 断裂让用户无法走通主流程，加上若干 UX 摩擦、安全技术债和工程清理项。本次规划覆盖 P0～P3 全部问题，目标是交付一个可以给真实用户试用的 v0.2 正式版本。

## Requirements

**P0 — 核心流程修复（先修才能给人用）**

- R1. 审稿页 `FACTS_KEYS` 改为从 `@51guapi/shared` 导入 `GOSSIP_FACT_KEYS`，显示「当事人/事件摘要/起因/经过/结果」而非旧漫画字段
- R2. 「批准」按钮废弃旧「填表发布」路径（`resolveAdminTabId` + `runBatch`），改为：批准 → 生成草稿 → 跳转 DraftPreview 供编辑/导出
- R3. 批准流程中若 LLM 调用失败，显示友好错误状态而非静默崩溃

**P1 — UX 摩擦消除**

- R4. App.tsx 主页对两条路径做视觉区分：「手动输入主题 → AI 草稿」与「爬取吃瓜素材 → 待审池」各有明确标签/说明
- R5. 新增站点时若站点域名不在渠道白名单，联动提示用户先添加渠道（或提供一键联动添加）
- R6. 待审详情展开区域显示提炼质量信号：`confidence` 分值、`extractionMode`（strict/fallback）、为 null 的字段高亮
- R7. Docker 生产部署路径修复：Dockerfile runtime 阶段改用 `pnpm deploy` 或 builder 阶段复制 prod 依赖，消除 `workspace:*` 协议解析错误

**P2 — 安全加固**

- R8. SSRF allowlist 写入接口增加二次验证（admin password）：仅凭普通 JWT 不应能新增渠道，防 XSS token 泄露后自扩渠道
- R9. 文档化 DNS rebinding 风险（socket-level IP pinning 难度高，本次以文档化 + 告警 log 为目标，不要求实现 pinning）

**P3 — 工程清理**

- R10. 包名去旧品牌「publisher」：`publisher-monorepo` → `51guapi-monorepo`，`publisher-backend` → `51guapi-backend`，`publisher-fill-assistant` → `51guapi-extension`；同步更新 CI `--filter` 引用和 README
- R11. `docs/plans/2026-06-17-005-feat-v0-2-release-readiness-plan.md` Unit 1-5 完成状态核查，将已落地的 checkbox 标为完成

## Success Criteria

- 用户从「发现吃瓜站点 → 爬取素材 → 待审 → 批准 → 导出 JSON/Markdown」全流程可无错走通
- 审稿页显示正确的吃瓜事实字段，confidence 信号可见
- Docker `docker compose up` 能启动后端（不再因 `workspace:*` 报错）
- 渠道新增需要管理员口令二次验证
- 包名不再包含「publisher」

## Scope Boundaries

- 不实现 socket-level IP pinning（DNS rebinding 完整修复）；本次只做文档化和告警
- 不新增爬取能力（JS 渲染、登录态页面）
- 不做用户账号体系或多用户支持
- 不做移动端适配

## Key Decisions

- **批准路径重设计**：选择「批准 → DraftPreview」而非「批准 → 直接导出」，保留人工二次编辑空间，符合「人工预览/编辑 → 导出」的产品定位
- **渠道联动提示**：选择「提示」而非「自动写入白名单」，维持 SSRF 守卫的 fail-closed 原则（爬取管线不可自触发 allowlist 写入）
- **包名重命名**：只改 package.json `name` 字段和 CI `--filter`，不改目录结构，降低风险

## Dependencies / Assumptions

- `GOSSIP_FACT_KEYS` 已在 `@51guapi/shared` 中定义（待核查）；若不存在需先在 shared 包中导出
- 批准后的草稿生成逻辑已有 `GENERATE_DRAFT` 消息路由，复用即可
