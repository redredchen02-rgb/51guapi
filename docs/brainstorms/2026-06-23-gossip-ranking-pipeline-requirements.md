# 吃瓜排行流水線

## Problem Frame

現有 pipeline 把各站抓回的資料堆進「待審選題」，使用者要一條一條手動看散亂的清單。這套流程缺少兩件事：一是把多站數據與社群熱搜做交叉比對找出「真正在爆的瓜」，二是把結果以加權排行表呈現，讓使用者能迅速選出最值得寫的選題。

目標：加入社群熱搜抓取、交集比對、加權評分，最終輸出一張直接可操作的「瓜排行表」，而非散亂原始資料。

## 資料流

```
用戶管理的吃瓜站點
        │  POST /api/v1/gossip/sites/:id/discover（已有）
        ▼
  站點話題（title, url, 時間）
        │
        ├──────────────── 模糊匹配 ───────────────┐
        │                                        │
社群熱搜平台（4 個）                           加權評分器
  • 微博熱搜                                    ↑
  • 抖音熱榜                          platform_count × W1（大瓜）
  • 百度熱搜                          heat_score × W2（流量瓜）
  • 小紅書熱點                        site_count × W3（站點覆蓋）
        │                            recency × W4（新瓜）
        ▼                                        │
  熱搜關鍵詞（keyword, heat, platform, ts）      │
        │                                        │
        └──────────────────────────────────────→ ▼
                                      瓜排行表（WebUI /gossip-rank）
                                      ┌──────────────────────────┐
                                      │ A 區：熱搜 ∩ 站點覆蓋    │
                                      ├──────────────────────────┤
                                      │ B 區：僅熱搜熱點（發現）  │
                                      └──────────────────────────┘
```

## Requirements

**站點管理 — 手動觸發爬取**

- R1. 吃瓜站點列表中，每個站點卡片有「立即爬取」按鈕，點擊後觸發該站點的內容抓取（串接現有 `POST /api/v1/gossip/sites/:id/discover`），並在按鈕上顯示進行中狀態。
- R2. 爬取完成後，站點卡片顯示「上次爬取時間」與「本次新增條目數」。

**社群熱搜抓取**

- R3. 後端新增熱搜抓取器，覆蓋 4 個平台：微博熱搜、抖音熱榜、百度熱搜、小紅書熱點。（注：各平台是否為 JS 渲染需在計畫階段 research，若需 headless browser 則分批上線。）
- R4. 每個平台的熱搜條目儲存：關鍵詞、熱度數值、平台標識、抓取時間戳。
- R5. 熱搜資料有 TTL（不超過 24 小時），避免過期資料影響排行。

**交叉比對與加權評分**

- R6. 系統把站點話題與熱搜關鍵詞做模糊匹配，判斷是否為同一瓜。
- R7. 每條話題計算加權分數，四個維度：
  - **大瓜**：同時出現在幾個熱搜平台（0–4，多平台 = 大瓜）
  - **流量瓜**：各平台熱度數值加總（正規化後加權）
  - **站點覆蓋**：有幾個用戶管理的站點也在報導這條瓜
  - **新瓜**：距首次發現的時間（越近得分越高，24 h 內為最高）
- R8. 加權公式的各維度比重可在後端設定調整（不需 UI，`.env` 或 config 即可）。

**瓜排行表（WebUI `/gossip-rank`）**

- R9. 瓜排行表分兩個區塊：
  - **A 區（精選）**：站點有報導 AND 至少 1 個熱搜平台出現 → 嚴格交集，依加權分排序
  - **B 區（發現）**：只在熱搜上出現、站點尚未覆蓋 → 依熱搜平台數 + 熱度排序
- R10. 表格欄位：排名、話題標題、加權分、平台 badge（顯示哪幾個平台有）、站點覆蓋數、首次發現時間、操作。
- R11. 頁面頂部有「立即抓取」按鈕，一次性觸發所有站點爬取 + 4 個熱搜平台抓取，完成後自動刷新排行表。
- R12. 操作欄提供三個動作：
  - **一鍵生成草稿**：以此話題為主題直接觸發 AI 草稿生成
  - **隱藏**：從表格移除（加入使用者黑名單，不再出現）
  - **來源連結**：展開看各站點原始 URL 及熱搜平台連結

**現有待審清單的關係**

- R13. 瓜排行表是新的入口，現有「待審選題」清單保留不動，兩者資料互通（從排行表生成的草稿同樣流入待審池）。
- R14. 排行表不顯示已被隱藏或已核准的選題。

## Success Criteria

- **開發環境（可測）**：以 fixture 熱搜資料 + fixture 站點話題，能正確產出 A 區交集條目並依分數排序。
- **手動 QA**：按下「立即抓取」，2 分鐘內排行表刷新，真實環境 A 區在有熱搜事件時出現至少 1 條交集話題。
- 從排行表點擊「一鍵生成草稿」，流程與現有從待審池生成草稿一致，草稿出現在草稿編輯器。
- 隱藏的條目不再出現，重新觸發抓取後仍保持隱藏。

## Scope Boundaries

- 不做定時自動抓取（手動觸發即可，排程可在之後加）。
- 不新增社群平台的貼文內容爬取，只抓熱搜關鍵詞列表（無正文）。
- 不改動現有「待審選題」頁面的 UI 及邏輯。
- 不做熱搜賬號登入，只抓取公開熱搜頁面。
- 不在 Chrome 擴充功能裡加新頁面（入口在 WebUI）。

## Key Decisions

- **分兩區塊而非嚴格交集**：嚴格交集會過濾掉潛力話題，B 區（僅熱搜）保留發現機會。
- **大瓜 = 多平台同時出現**：平台覆蓋數最直觀，不依賴不穩定的貼文互動數。
- **加權參數不做 UI**：早期用 config/env 調整即可，不過設計 UI。
- **熱搜抓取為 public 頁面**：不處理登入態，降低維護成本與帳號風險。
