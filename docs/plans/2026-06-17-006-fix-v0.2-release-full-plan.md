# fix: v0.2 全面治理

## Overview

修复审稿流程两处 P0 断裂（字段显示错误 + 批准按钮报错），补上质量信号展示、站点/渠道联动提示、Docker 部署修复、渠道写入二次验证，最后做包名重命名。目标是交付可给真实用户试用的 v0.2 正式版。

## Problem Frame

51guapi 功能骨架完整，但两处断裂让用户无法走通主流程：审稿页显示旧漫画字段（当事人/事件经过不出现），批准按钮调旧填表路径（找目标页标签）报错。加上站点配置失败率高、Docker 不可用、渠道安全加固缺口等问题，合计 P0-P3 共 10 项。

## Requirements Trace

- R1. 审稿页显示吃瓜事实字段（GOSSIP_FACT_KEYS）
- R2. 批准 → 生成草稿 → DraftPreview（废弃 RUN_BATCH 路径）
- R3. 批准失败时显示友好错误状态
- R4. App.tsx 主页两条路径视觉区分
- R5. 新增站点时若域名不在渠道白名单，联动提示
- R6. 待审详情显示 confidence + null 字段高亮（本次不含 extractionMode，需后端独立改动）
- R7. Docker runtime 阶段修复 workspace:* 协议报错
- R8. 渠道（channels 白名单，非 gossip-sites）新增增加管理员口令二次验证
- ~~R9. DNS rebinding 文档化~~ — 已覆盖于 CLAUDE.md，不建实施单元
- R10. 包名去旧品牌 publisher，同步 CI filter 引用

## Scope Boundaries

- 不实现 socket-level IP pinning（DNS rebinding 完整修复）
- R6 中 extractionMode 显示依赖后端 pending-store 改动，不在本计划内
- 不新增爬取能力（JS 渲染、登录态页面）
- 不做用户账号体系或多用户支持
- R9（DNS rebinding 文档化）已在 CLAUDE.md 覆盖，不单独建实施单元
- R2 废弃 RUN_BATCH 在审稿批准路径；若存在其他 RUN_BATCH 触发点，需单独审查，不在本计划改动范围内

### Institutional Learnings

- **包名重命名 CI 静默失败**（`vitest-excludes-dist-phantom-backend-p0-2026-06-15`）：`pnpm --filter` 取自 `package.json` 的 `name` 字段，改完后必须同步所有 `--filter` 引用，否则 CI 全绿但什么都没跑。改完后实跑验证 `0 tests` vs 有具体数字。
- **fail-closed 三问**（`fixture-secret-gate-false-green-relative-path-2026-06-15`）：任何安全闸门，空输入必须全拒，不依赖调用方 cwd。
- **background.ts 消息处理**（`incremental-pr-adversarial-verification-2026-06-15`）：先检查无变化 → 直接返回同一引用，有变化 → 单次原子持久化。

## Key Technical Decisions

- **R2 批准路径**：改发 `GENERATE_DRAFT` 消息，**扩展 payload 格式为 `{ prompt: string, facts: GossipFactsBlock }`**，同步修改 background.ts `handleGenerate` 签名接受 facts 并传入 assemblePrompt。`PendingTopicsView` 新增 `onDraftReady: (draft: ContentDraft) => void` prop，App.tsx 收到后 `setDraft(draft); setMode("draft")`。
- **R2 status 写入时序**：`updatePendingStatus(id, "approved")` 移到 draft 成功返回后执行，LLM 失败时 topic status 保持 pending，可重试，不产生孤儿状态。
- **R5 联动提示方式**：前端在 `handleAdd` 提交前 `fetchChannels()`，用 `new URL(listUrl).hostname` 比对，不在列表时渲染黄色提示框但不硬拦截。不自动写白名单（维持 SSRF fail-closed 原则）。
- **R6 范围**：`confidence` 直接展示（字段已在 `PendingTopic` interface），`extractionMode` 依赖后端 pending-store 改动，本次不做，记为 deferred。
- **R7 修法**：选 builder 阶段 COPY node_modules（最简单，零额外工具依赖）而非 pnpm deploy 或 tarball，因为 pnpm deploy 在 workspace 拓扑序上有额外约束。
- **R8 对象**：`POST /api/v1/channels`（渠道白名单）而非 gossip-sites；gossp-sites 新增不需要 admin 权限，保持现状。GossipView 已在上一轮 PR 加入管理员口令字段，核查是否已对接 `adminPassword` 字段传参。
- **R10 顺序**：先改 4 个 package.json name → 再全局替换 CI/scripts 的 `--filter` 引用 → Dockerfile LABEL 同步，一次提交，避免中间态 CI 失效。

## High-Level Technical Design

> *这是方向性设计示意，不是实施代码。实施者以此为上下文，不要逐字复制。*

**R2 批准流程重设计（核心改动）：**

```
用户点击「批准」
    ↓
PendingTopicsView.handleApproveSelected()
    ↓
① updatePendingStatus(id, "approved")  ← 保留
② 废弃 runBatch() + RUN_BATCH 消息
③ 改发 chrome.runtime.sendMessage({ type: "GENERATE_DRAFT", prompt: ... })
    ↓
background.ts.handleGenerate(prompt)
    → 调 LLM → 返回 { draft: ContentDraft }
    ↓
PendingTopicsView.props.onDraftReady(draft)
    ↓
App.tsx: setDraft(draft); setMode("draft")
    ↓
DraftPreview 渲染（带 facts 传入）
```

**R5 站点/渠道联动提示流程：**

```
GossipView.handleAdd()
    ↓
new URL(listUrl).hostname → hostname
    ↓
fetchChannels() → channels[]
    ↓
channels.some(c => c.hostname === hostname)?
    ├─ true  → 直接调 createGossipSite()
    └─ false → 显示提示框
               「该域名未在渠道白名单，爬取将被 SSRF 守卫拒绝。
                 建议先在渠道管理中添加 {hostname}。[仍然继续] [取消]」
               用户点「仍然继续」→ 调 createGossipSite()
```

## System-Wide Impact

- **Interaction graph:** R2 改动 background.ts 消息路由，`RUN_BATCH` 在 PendingTopicsView 批准路径中不再被触发；确认其他触发点（若有）不受影响
- **Error propagation:** R3 错误在 PendingTopicsView 层处理，不向上冒泡到 App.tsx
- **State lifecycle risks:** R2 中批准后 `topic.status` 变为 "approved"，若 LLM 失败（R3）用户可重试，status 已改为 approved，重试时注意幂等（`updatePendingStatus` 再次调用应不报错）
- **API surface parity:** R8 的 channel 二次验证只影响扩展前端，后端路由不变
- **Integration coverage:** R2 的完整链路（批准 → GENERATE_DRAFT → DraftPreview 渲染）需 integration 测试覆盖，mock sendMessage 不足以证明端到端链路
- **Unchanged invariants:** 导出路径（JSON/Markdown）、SSRF 守卫 fail-closed 行为、LLM 只写口吻散文槽位，均不受本计划影响

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| GOSSIP_FACT_KEYS 是普通数组而非 as const tuple，类型索引可能报错 | Unit 1 实施时用 `as Array<keyof GossipFactsBlock>` 断言；参考 export.ts 的 `facts?.[k] ?? ""` 模式 |
| **[P0] handleGenerate 签名不接受 facts**：现有 `handleGenerate(prompt: string)` 无 facts 参数，批准流程事实注入会静默失败 | Unit 2 实施前确认：若 handleGenerate 不接受 facts，需扩展为 `{ prompt, facts }` 对象格式（或改走 RUN_BATCH 单条 topic 路径）；不可跳过 |
| **[P0] R2 status 先写入导致孤儿状态**：LLM 失败后 topic.status 已是 approved 但无 draft | 将 `updatePendingStatus` 移到 draft 成功返回后执行；或失败时以 `updatePendingStatus(id, "pending")` 回滚 |
| **[P0] Docker pnpm symlink 断链**：pnpm node_modules 使用 symlink 指向 virtual store，COPY 后符号链接目标在 runtime image 不存在 | Unit 7 实施前验证 builder stage 是否用 `--shamefully-hoist` 或 `node-linker=hoisted`；若用默认 symlink 模式，改用 `pnpm deploy --filter publisher-backend` 方案 |
| R10 包名重命名后 CI 静默跳过 | Unit 9 完成后立即实跑 `--filter` 命令确认有输出，不是 0 tests；同步更新 README 和 CLAUDE.md 中的 publisher-* 引用 |
| Docker COPY shared/dist 若 builder 未生成 | Unit 7 实施时确认 builder stage 的 build step 覆盖 @51guapi/shared |
| Chrome MV3 onMessage async 返回机制 | Unit 2 实施前确认 background.ts onMessage listener 已 `return true` + sendResponse 模式，否则 sendMessage 回调在 MV3 service worker 中可能丢失 |
